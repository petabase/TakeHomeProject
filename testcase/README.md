# Test Cases

Ready-to-upload test data for the TTB Label Verifier, organized by how you'd actually exercise the app: a single-label check and a full batch run.

## Folder layout

```
testcase/
├── SingleLabel/
│   ├── WheatBeer.png                # one label image (manually created)
│   ├── wheat-beer-single-meta.csv   # its matching 1-row application data
│   └── wheat-beer-single-results.csv  # expected output from a single-label run
└── batchLabels/
    ├── ttb_real_labels_images.zip   # 14 real TTB label images
    ├── batch-metadata.csv           # application data for all 14 images
    └── batch-results.csv            # expected/ground-truth output for the batch
```

## How to run

### Single label

`SingleLabel/` is a manually created example for quickly sanity-checking the single-label path without running a full batch.

1. Start the app (`./run.sh` or `make run` from the repo root)
2. Go to `http://localhost:8000`
3. Upload `SingleLabel/wheat-beer-single-meta.csv` as the metadata CSV
4. Upload `SingleLabel/WheatBeer.png` as the image
5. Click **Verify**
6. Compare the result against `SingleLabel/wheat-beer-single-results.csv`

### Batch

`batchLabels/` is generated by [`testcase-generator/extract_real_labels.py`](../testcase-generator/README.md) from TTB's own official labeling guide PDF. See that folder's README for how to regenerate it if needed.

1. Start the app
2. Go to `http://localhost:8000`
3. Upload `batchLabels/batch-metadata.csv` as the metadata CSV
4. Switch to **Upload a .zip** mode and select `batchLabels/ttb_real_labels_images.zip`
5. Click **Verify**
6. Export the results as CSV and compare against `batchLabels/batch-results.csv`

## Where this data comes from

**`batchLabels/`** is sourced from TTB's own official labeling guide — see [`testcase-generator/README.md`](../testcase-generator/README.md) for the full explanation of how those images were extracted, which scenarios were selected, and which ones have deliberate angle/blur/glare distortion applied. `batch-results.csv` is the expected/ground-truth output (columns: `filename`, `expected_outcome`, `distortion`, `source`, `reason`) — if a real run disagrees with it, that's either a regression or a Gemini Vision reading variance worth investigating, not necessarily a bug, since vision model output can vary slightly run to run on borderline cases.

**`SingleLabel/`** is manually created — the image and CSV were hand-crafted to represent a typical single-label check. No script generates it.

## A note on ERROR rows in batch results — this is not a bug

If some rows show `ERROR` status with a "quota exceeded" message, the Gemini free-tier rate limit (15 requests/minute) was hit, not a code defect.

The app fails fast on a 429 rather than retrying silently — retrying would stall the UI for 60+ seconds with no feedback. The affected rows are flagged with a plain-English explanation.

**To resolve:** wait 60 seconds and re-run the batch, or split it into smaller chunks. The `MIN_REQUEST_DELAY_SECONDS = 4.5` spacing in `verifier.py` keeps most runs under the limit, but if the quota was already partially consumed before the batch started (e.g. from a prior run in the same minute), some rows may still hit it. To remove this limit entirely, add a billing method at [ai.dev/rate-limit](https://ai.dev/rate-limit).

## A note on PASS vs NEEDS REVIEW for high-confidence formatting differences

A field that matches only after normalizing case/punctuation (e.g. label shows `EXAMPLE BREWING CO.`, application says `Example Brewing Co.`) is not automatically a hard fail — per Dave's interview feedback, this kind of formatting-only difference shouldn't stop an otherwise-correct label.

The rule: a normalized-match field with **confidence ≥ 75%** auto-passes into an overall PASS. Below 75% — typically a partial/substring match where the comparison itself is genuinely uncertain — the field still routes to NEEDS REVIEW. This threshold lives in `app/verifier.py` as `AUTO_PASS_CONFIDENCE_THRESHOLD`.

If you see a label pass in a real run despite a "Likely Match" field, check the field's `confidence_pct` in the detail view — anything 75%+ on a normalized match is expected correct behavior, not a bug.
